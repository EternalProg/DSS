# 3. Архітектура системи

## 3.1. Загальний опис

Система підтримки прийняття рішень реалізована як клієнт-серверний вебдодаток із класичною трирівневою архітектурою:

1. клієнтський рівень (інтерфейс);
2. серверний рівень (REST API та бізнес-логіка);
3. рівень даних (MongoDB).

Такий підхід забезпечує чітке розділення відповідальності, спрощує масштабування та дозволяє незалежно розвивати UI, API і модуль обчислень.

**Клієнтський рівень** (`public/`) - односторінковий вебінтерфейс, який завантажується як статичні файли (`index.html`, `styles.css`, `app.js`).

UI відповідає за:
- введення даних (альтернативи, критерії, оцінки, ваги);
- відображення таблиць, матриці й аналітичних результатів;
- ініціювання обчислень і побудову графіків через Chart.js;
- взаємодію із сервером через Fetch API у форматі JSON.

**Серверний рівень** реалізовано на Node.js + Express.

Сервер виконує:
- приймання HTTP-запитів;
- валідацію вхідних даних;
- CRUD-операції для доменних сутностей;
- запуск аналітичних стратегій і формування рекомендацій;
- доступ до MongoDB через сервісний шар.

Архітектурно сервер поділено на два підрівні:

1) **Контролери (`routes/`)** - маршрутизація та REST API:
- `alternatives.js`, `criteria.js`, `evaluations.js` - CRUD-операції;
- `matrix.js` - формування матриці оцінювання;
- `analytics.js` - розрахунок інтегральних оцінок, ранжування та рекомендації.

2) **Сервіси (`services/`)** - ізольована бізнес-логіка:
- `db.js` - підключення до MongoDB, ініціалізація індексів, доступ до колекцій;
- `analytics.js` - реалізація алгоритмів обчислення (обережна, адитивна, мультиплікативна стратегії) і генерація пояснень.

**Рівень даних** - MongoDB, де зберігаються:
- альтернативи (платформи);
- критерії (тип `maximize`/`minimize` і вага);
- оцінки (зв'язок альтернативи й критерію + числове значення).

## 3.2. Потік даних у системі

Типовий сценарій роботи:

1. Користувач додає або редагує дані в інтерфейсі.
2. Клієнт надсилає HTTP-запит до API у форматі JSON.
3. Сервер валідовує запит, звертається до MongoDB та виконує потрібну операцію.
4. API повертає результат клієнту.
5. UI оновлює таблиці, матрицю, блок рекомендацій і графіки.

Завдяки цьому архітектура залишається читабельною, модульною і придатною до подальшого розширення.

## 3.5. Принципи організації коду

### Розділення відповідальності (Separation of Concerns)

1. `routes/` - обробка HTTP-запитів і відповідей.
2. `services/` - бізнес-логіка, незалежна від транспортного шару.
3. `public/` - презентаційний рівень (UI).

### Модульність

1. Кожна сутність (альтернативи, критерії, оцінки) має окремий роутер.
2. Аналітичний модуль винесено в окремий сервіс.
3. Компоненти інтерфейсу та API-ендпоінти слабо зв'язані між собою.

### Розширюваність

1. Нові стратегії додаються в `services/analytics.js`.
2. Нові ендпоінти додаються як окремі роутери в `routes/`.
3. UI-блоки можна розвивати незалежно (таблиці, матриця, графіки, пояснення).

# 4. Рівень даних (MongoDB)

## 4.1. Структура бази даних

У системі використовується 3 основні колекції: `alternatives`, `criteria`, `evaluations`.

### `alternatives`
- призначення: зберігання платформ для порівняння;
- ключові поля: `name`, `description`, `createdAt`, `updatedAt`;
- обмеження: унікальний індекс на `name`.

### `criteria`
- призначення: зберігання критеріїв оцінювання;
- ключові поля: `name`, `type` (`maximize`/`minimize`), `description`, `weight`;
- обмеження: унікальний індекс на `name`.

### `evaluations`
- призначення: значення оцінок для пари "альтернатива - критерій";
- ключові поля: `alternativeId`, `criterionId`, `value`, `createdAt`, `updatedAt`;
- обмеження: складений унікальний індекс `{ alternativeId, criterionId }`.

# 6. Аналітичний модуль

## 6.1. Архітектура модуля

Аналітичний модуль реалізовано у файлі `services/analytics.js`. Він містить:
- три стратегії обчислення інтегральної оцінки;
- функцію генерації пояснень для користувача;
- головну функцію аналізу з формуванням рекомендації.

## 6.2. Стратегії обчислення

### 6.2.1. Обережна (Cautious) стратегія

**Формула:** `Q(A_i) = min(w_j * x_ij)`

**Суть:** інтегральна оцінка альтернативи визначається як мінімальне значення серед усіх зважених оцінок. Це песимістичний підхід, що концентрується на найслабшому місці альтернативи.

**Коли використовувати:**
- для критичних систем, де неприпустимі слабкі місця;
- коли існують регуляторні або безпекові обмеження;
- у ситуаціях високого ризику.

### 6.2.2. Адитивна (Additive) стратегія

**Формула:** `Q(A_i) = Σ(w_j * x_ij)`

**Суть:** інтегральна оцінка обчислюється як сума всіх зважених оцінок. Критерії можуть частково компенсувати один одного.

**Коли використовувати:**
- коли потрібен збалансований компроміс між факторами;
- для загальних задач вибору без жорстких порогів;
- коли важлива прозора й зрозуміла інтерпретація результату.

### 6.2.3. Мультиплікативна (Multiplicative) стратегія

**Формула:** `Q(A_i) = Π(x_ij^w_j)`

**Суть:** інтегральна оцінка - добуток значень оцінок, піднесених до степенів ваг. Низькі значення мають сильніший негативний вплив на результат.

**Коли використовувати:**
- коли критерії взаємопов'язані та мають мультиплікативну природу;
- коли "провал" за одним критерієм суттєво знижує якість альтернативи;
- для задач, де потрібне підсилене покарання слабких показників.

## 6.3. Рекомендація стратегії

Для задачі вибору платформи дистанційного навчання як базову рекомендовано **адитивну** стратегію (із можливістю паралельно рахувати й інші):

1. критерії можуть компенсувати один одного;
2. немає єдиного критичного фактора, який робить альтернативу абсолютно неприйнятною;
3. університет зазвичай приймає компромісне рішення між вартістю, функціональністю і зручністю;
4. метод є найбільш поширеним і зрозумілим для пояснення стейкхолдерам.

# 7. Клієнтський рівень (UI)

## 7.1. Загальний опис інтерфейсу

Клієнтська частина реалізована як односторінковий вебдодаток з адаптивним дизайном. Інтерфейс поділено на логічні секції:

1. Заголовок - назва системи та швидкий доступ до аналітики.
2. Альтернативи - керування платформами для порівняння.
3. Критерії - керування факторами оцінювання.
4. Оцінки - матриця оцінювання з inline-редагуванням.
5. Ваги критеріїв - налаштування важливості (1-10).
6. Аналіз та рекомендації - вибір стратегій, результати, візуалізація й пояснення.

## 7.2. Секція "Альтернативи"

Дозволяє:
- додавати нові альтернативи (назва, опис);
- редагувати наявні альтернативи;
- видаляти альтернативи з підтвердженням.

## 7.3. Секція "Критерії"

Дозволяє:
- додавати нові критерії (назва, тип `maximize`/`minimize`, опис);
- редагувати критерії;
- видаляти критерії з підтвердженням.

## 7.4. Секція "Оцінки"

Реалізовано два способи введення оцінок:

1. **Через форму** - вибір альтернативи, критерію та числового значення.
2. **Inline-редагування** - подвійний клік на комірці матриці для швидкого редагування.

## 7.5. Секція "Ваги критеріїв"

Відображає всі критерії з можливістю встановлення ваги від 1 до 10:
- назва критерію та його тип;
- поле вводу ваги;
- автоматичне збереження через `PATCH /api/criteria/:id/weight`.

## 7.6. Секція "Аналіз та рекомендації"

### 7.6.1. Вибір стратегій

Користувач може обрати одну або кілька стратегій:
- Обережна;
- Адитивна (позначена як рекомендована);
- Мультиплікативна.

Кожна стратегія має:
- назву;
- математичну формулу;
- підказку з поясненням.

### 7.6.2. Блок рекомендації

Після розрахунку відображається:
- назва рекомендованої альтернативи;
- загальний бал;
- причина вибору стратегії.

### 7.6.3. Таблиця результатів

Показує ранжування альтернатив за кожною обраною стратегією:
- позиція в рейтингу (з бейджами для 1-3 місця);
- числовий бал.

### 7.6.4. Візуалізація

Реалізовано чотири графіки на базі Chart.js:

1. Bar chart "Обережна".
2. Bar chart "Адитивна".
3. Bar chart "Мультиплікативна" (для великих чисел автоматично використовується логарифмічне відображення).
4. Radar chart профілів альтернатив за критеріями.

### 7.6.5. Пояснення результатів

Для кожної обраної стратегії формується текстове пояснення:
- переможець і його бал;
- фактори перемоги (топ критерії);
- порівняння з другим місцем.
